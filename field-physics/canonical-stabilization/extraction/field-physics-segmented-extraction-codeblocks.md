---

# 1. Revised Source Architecture

The full ten-file stabilization package remains in the repository as the **verification and adjudication apparatus**.

It is not uploaded to every Liora extraction session.

Each ordinary extraction session receives only:

1. `field-physics-foundational-extraction-capsule.md`
2. one numbered foundational-chat segment
3. the cumulative findings ledger for that foundational document
4. the segment manifest, where separate from the findings ledger

No FP file is uploaded unless Liora requests one because exact present wording is necessary.

No full ledger, full synonym concordance, full glyph concordance, or full numerical register is uploaded by default.

---

# 2. Why the Ten Files Still Matter

The ten files are not discarded.

They remain necessary for:

- Lumen's current-corpus verification
- normalization of `MAP-`, `GLY-`, `SYN-`, `NUM-`, and audit-flag references
- Aelion's adjudication
- later conceptual review
- final adversarial audit
- repository provenance

The compact capsule carries only the context Liora needs for first-pass extraction.

---

# 3. Segment Size

Split each foundational chat by **word count**, not by nominal page count.

Recommended target:

- **15,000–20,000 words per segment**
- acceptable maximum: **25,000 words** where a natural section boundary requires it
- do not split in the middle of one tightly connected exchange where a small extension preserves context

A 350-page development record will probably require approximately **seven to ten segments**, depending on page density.

Assign segment IDs:

- `CD1-S01`, `CD1-S02`, ...
- `CD2-S01`, `CD2-S02`, ...

Each segment file should begin with:

- source document name
- segment ID
- original page or message range
- date range, where known
- previous-segment boundary note
- next-segment boundary note, where known

---

# 4. One Fresh Chat per Segment

Use a fresh high-effort Liora chat for each segment.

This is intentional. Continuity is carried through files rather than conversational memory.

## Segment 1 Upload Order

1. extraction capsule
2. findings-ledger template or empty `FIND-CD1` file
3. `CD1-S01`

## Segment 2 and Later Upload Order

1. extraction capsule
2. latest cumulative `FIND-CD1` file
3. current segment

Do not upload earlier source segments again.

The cumulative findings ledger carries:

- all findings already extracted
- unresolved duplicate families
- continuing open threads
- the next finding number
- the last completed segment

---

# 5. First Instruction in Every Segment Chat

Use this instruction:

> Continue the Field Physics foundational extraction under the attached capsule. Read only the attached source segment. Preserve the continuous `FIND-CD1-###` numbering in the cumulative findings ledger. Extract candidate material without adjudicating open questions or drafting restoration prose. Update the cumulative findings file and return it in a fenced Markdown code block.

For Chat Document 2, replace `CD1` with `CD2` and attach the completed CD1 findings file as a deduplication reference only when context permits. If that file becomes too large, attach a compact CD1 deduplication index instead.

---

# 6. Continuous Finding Numbers

Finding numbers do not restart across segments or chats.

Example:

- `CD1-S01` ends with `FIND-CD1-018`
- `CD1-S02` begins with `FIND-CD1-019`

If a later segment expands an earlier finding, do not create a duplicate automatically.

Update the earlier finding with:

- new source location
- expanded evidence
- chronology
- changed confidence

Create a new finding only where the later passage performs a materially distinct structural function.

---

# 7. Current-Corpus Comparison During Extraction

The capsule provides sufficient orientation for provisional judgments.

Use these labels:

- already preserved
- preserved but weakened
- materially omitted
- renamed or superseded
- current coverage uncertain

Where exact wording determines the result:

1. mark `Current Coverage: Verification required`
2. name the likely FP file
3. request that one FP file at the end of the segment

Do not halt the entire segment for a single uncertain comparison.

---

# 8. Segment-End Output

At the end of every segment, Liora returns:

1. the updated cumulative findings file
2. a short segment completion note stating:
   - segment ID
   - source range completed
   - number of new findings
   - findings materially expanded
   - duplicate families identified
   - continuing threads
   - requested FP files, if any
3. the next finding number

The updated findings file is the only mandatory carryover into the next segment chat.

---

# 9. Completion of Chat Document 1

After the final CD1 segment, open a separate consolidation chat with:

1. extraction capsule
2. completed `FIND-CD1` file
3. no source segments unless a disputed finding requires exact reinspection

Liora then performs only:

- duplicate consolidation
- chronology cleanup
- source-location completeness check
- open-thread closure check
- candidate-family grouping
- creation of a compact CD1 deduplication index for Chat Document 2

Liora does not adjudicate restoration.

---

# 10. Chat Document 2

Repeat the same segmented architecture.

Each CD2 segment session receives:

1. extraction capsule
2. cumulative `FIND-CD2` file
3. compact CD1 deduplication index
4. current CD2 source segment

The full CD1 findings ledger should be uploaded only if it remains context-safe. Otherwise the deduplication index carries:

- finding ID
- working name
- source concept
- current classification
- aliases
- duplicate-search terms

---

# 11. Lumen Verification

After both extraction ledgers are complete, Lumen receives:

- full ten-file stabilization package
- completed `FIND-CD1`
- completed `FIND-CD2`
- source passages cited by serious findings
- individual foundational segments as needed

Lumen then:

- normalizes stable references
- checks current FP coverage
- searches synonyms and renamed concepts
- identifies missed candidates
- tests commentary contra primary prose
- attaches missing conceptual dependencies
- prepares the adjudication ledger for Aelion

---

# 12. Governing Principle

Long-source extraction must be **stateful through artifacts, not dependent upon one overloaded conversation**.

The cumulative findings ledger is the memory.

The extraction capsule is the law.

The source segment is the only material under immediate review.
